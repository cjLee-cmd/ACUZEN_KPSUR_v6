1. Stage 2 - 사용자 입력 정보 불일치
  - 사용자 입력시 '약품명'을 선택하면 따라오는 정보들. 약품명에 대한 DB가 별도로 있어야 함. 이 정보 DB 테이블을 만들고 입력하는 UI 추가.
  - 11개 데이터 모두 약품명에 종속됨. 따라서 DB에 모두 별도로 미리 입력되어 있어야 함.
  
    1. 사양서의 11개 변수가 맞습니까? (워크플로우에 누락된 것인가요?) 
        - 다 있어야 함.
    2. 자동 계산 로직이 실제 업무 규칙입니까?
        - 규칙 임.
    3. 날짜 검증 순서 (CS5 < CS4 < CS6 < CS13)가 정확합니까?
        - 맞음

2. RAW ID - 필수/선택 구분 불명확
  - 워크플로우에 있는 문서가 모두 있어야 함.

3. 데이터 추출 - 충돌 해결 정책
  - 충돌괴거나 모순되는 데이터는 무조건 사용자에게 물어봐서 확정해야 함.

4.QC - 사용할 LLM 모델
    1. QC에 Flash-Thinking 모델 사용 확정입니까? 
        - 초기값을 그 모델로 설정하고, 변경 가능하도록 UI를 구성할 것.
    2. "Ultrathink..." 지침도 프롬프트에 포함합니까? (모델이 자체적으로 사고하므로 중복일 수 있음)
        - 중요한 질의는 포함할 것.
    3. 예상 소요 시간과 비용은 수용 가능합니까?
        - 진행해 보고 나중에 결정함. LLM 질의 데이터 저장할 때, 상위에 데이터량과 비용에 대한 데이터 추가.

5. ⚠️ 저장 위치 불일치 (매우 중요)
    1. 워크플로우의 로컬 경로는 테스트 모드 전용입니까?
        - 테스트모드 전용임. 모든 데이터는 DB에 저장 되어야 함.
    2. 실제 배포에서는 Supabase Storage만 사용합니까?
        - supabase storage 만 사용 함.
    3. 파일 저장 정책:
        - DB (필수) + Storage (백업)?
            - 이 방식 적용.
        - Storage (필수) + DB (메타데이터만)?
            - 적용하지 않음.
